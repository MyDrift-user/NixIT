# Secrets Management

## Overview

NixIT uses [sops-nix](https://github.com/Mic92/sops-nix) for managing secrets. Secrets are encrypted with age keys derived from each machine's SSH host key, stored encrypted in Git, and decrypted automatically at system activation time.

## How It Works

```
┌──────────┐    encrypted     ┌──────────┐    decrypted     ┌──────────┐
│ secrets/ │ ──── git ────→   │ /nix/    │ ── activation ─→ │ /run/    │
│ *.yaml   │    (safe)        │ store    │    (boot)        │ secrets/ │
└──────────┘                  └──────────┘                  └──────────┘
```

- Secret files (`secrets/*.yaml`) are encrypted and safe to commit
- At build time, sops-nix includes the encrypted files in the Nix store
- At activation time (boot/rebuild), they're decrypted using the machine's age key
- Decrypted secrets appear at `/run/secrets/<name>` with correct ownership/permissions

## This fleet is already bootstrapped

Real secrets were generated by `scripts/gen-secrets.sh` (+ `gen-kuze-pw.sh` for the
per-host `kuze` passwords). The result:

- **Plaintext source of truth — OUTSIDE this repo** in `../mdl-infra/deployments/`:
  - `_admin/age-key.txt` — your master age key (decrypts/edits all secrets)
  - `_admin/kuze-passwords.txt` — the per-host `kuze` login passwords
  - `<host>/ssh_host_ed25519_key(.pub)` — each host's pre-generated SSH key
    (seeded at install so its age identity is known up front)
  - `<host>/info.md` and `SECRETS.md` — human-readable dumps
  - **Back this folder up and keep it private — it is the keys to everything.**
- **In the repo (encrypted, safe to commit):** `.sops.yaml` (real age recipients for
  every host + admin) and `secrets/*.yaml` (`ENC[AES256_GCM…]`).

### Editing secrets

```bash
export SOPS_AGE_KEY_FILE=../mdl-infra/deployments/_admin/age-key.txt
sops secrets/common.yaml      # opens decrypted in $EDITOR; re-encrypts on save
```

### Still placeholders (external — can't be generated)

Set these before the relevant host goes live: every `newt/<host>` (Pangolin tunnel
creds), `fortivpn/config` (the FortiGate), and `moodleng` `OPENAI_API_KEY` (optional).

### Per-host `kuze` password

The admin user's password is **per host**: sops key `kuze/password/<hostname>`. The
user modules reference `kuze/password/${config.networking.hostName}` so each machine
picks its own. Plaintext is in `_admin/kuze-passwords.txt`.

### Adding a NEW machine

Easiest: re-run `scripts/gen-secrets.sh` (idempotent — it keeps existing keys and
adds any new host). Or manually:

1. Pre-generate the host key + derive its age pubkey:
   ```bash
   ssh-keygen -t ed25519 -N "" -f ../mdl-infra/deployments/<host>/ssh_host_ed25519_key
   nix run nixpkgs#ssh-to-age -- -i ../mdl-infra/deployments/<host>/ssh_host_ed25519_key.pub
   ```
2. Add `- &<host> age1…` to `.sops.yaml` keys + reference it under the matching
   `creation_rules`.
3. Re-encrypt so the new host can read existing secrets:
   ```bash
   sops updatekeys secrets/common.yaml && sops updatekeys secrets/ad-dc.yaml
   ```
4. `install-host.sh` seeds that pre-generated key automatically.

## Using Secrets in NixOS Modules

### Basic Usage

```nix
{ config, ... }: {
  sops.secrets."my-app/api-key" = {
    sopsFile = ../../../secrets/common.yaml;
  };

  # Secret is available at: config.sops.secrets."my-app/api-key".path
  # Which resolves to: /run/secrets/my-app/api-key
}
```

### With Ownership

```nix
sops.secrets."my-app/api-key" = {
  sopsFile = ../../../secrets/common.yaml;
  owner = "myapp";     # Unix user that owns the file
  group = "myapp";     # Unix group
  mode = "0400";       # Permissions (default: 0400)
};
```

### For Services with DynamicUser

Services using systemd `DynamicUser=true` (like step-ca) don't have a static user. Use world-readable mode or `LoadCredential`:

```nix
sops.secrets."ca/password" = {
  sopsFile = ../../../secrets/ad-dc.yaml;
  mode = "0444";       # Readable by the dynamic user
};
```

### Restart Services on Secret Change

```nix
sops.secrets."my-secret" = {
  sopsFile = ../../../secrets/common.yaml;
  restartUnits = [ "my-service.service" ];
};
```

## Key Rotation

To rotate a secret:

1. Edit the secrets file: `sops secrets/common.yaml`
2. Change the value
3. Commit and push
4. Rebuild affected machines: `nix-rebuild`

To rotate machine keys (after SSH key regeneration):

1. Get the new age key from the machine
2. Update `.sops.yaml` with the new key
3. Re-encrypt: `sops updatekeys secrets/*.yaml`
4. Commit and push

## Troubleshooting

### "Failed to decrypt" during rebuild

The machine's age key doesn't match any key in `.sops.yaml`. Re-run the setup steps above.

### Secret file is empty at /run/secrets/

Check `journalctl -u sops-nix` for decryption errors. Common causes:
- The secret key name doesn't match between the YAML file and your NixOS config
- The YAML file has syntax errors

### "No matching creation rule" when editing

Your `.sops.yaml` `path_regex` doesn't match the file path. Check the regex.
